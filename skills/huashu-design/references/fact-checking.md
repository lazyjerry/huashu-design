# Fact Checking：事實驗證先於假設

任何涉及具體產品、技術、事件、人物、版本號、發布狀態、規格參數的斷言，第一步先查，不靠記憶。

## 何時強制啟動

- 使用者提到你不熟悉或不確定的具體產品名
- 涉及 2024 年後的發布時間線、版本號、規格
- 你腦中出現「我記得」「應該還沒發布」「可能不存在」
- 使用者要求替特定產品或公司做設計物料

## 硬流程

1. 先做 WebSearch：產品名 + 最新時間詞，例如 `2026 latest`、`launch date`、`release`、`specs`
2. 讀 1-3 條權威來源，確認存在性、發布狀態、最新版本、關鍵規格
3. 把確認過的事實寫進專案的 `product-facts.md`
4. 搜不到或結果互相矛盾時，停下來問使用者，不自行補完

## 禁止句式

- 「我記得 X 還沒發布」
- 「X 目前是 vN」
- 「X 可能不存在」
- 「據我所知 X 的規格是...」

改成：

- 「我先查 X 最新狀態」
- 「權威來源顯示 X 是...」

## 與資產流程的關係

這一步先於 `references/asset-protocol.md`。先確認品牌或產品存在、已發布、版本正確，才談 logo、產品圖、UI 截圖、色值。

## `product-facts.md` 模板

```markdown
# <Product> · Product Facts
> 採集日期：YYYY-MM-DD
> 查核來源：<官方網站 / 新聞稿 / 文件 / 商店頁>

- 存在性：<已確認 / 未確認>
- 發布狀態：<已發布 / 未發布 / 測試中>
- 最新版本：<版本號或型號>
- 關鍵規格：
  - <規格 1>
  - <規格 2>
  - <規格 3>
- 備註：<仍待確認的點>
```

## 真實反例

- 使用者要做 DJI Pocket 4 發布動畫
- 若憑記憶說「還沒發布」就會把官方發布物料誤做成概念片
- 正確做法是先查，再決定要不要做真產品發布語氣

## 成本觀念

WebSearch 幾秒，返工常是 1-2 小時。這是最便宜的保命步驟。
